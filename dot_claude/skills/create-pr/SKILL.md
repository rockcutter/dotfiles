# PR作成

PRの作成から関連issueへの記録までを一貫して行う。

## 全体の流れ

1. 前提確認（ブランチ・コミット状態・既存PR）
2. 関連issueの特定
3. push
4. draft PRの作成
5. 親issueのdescriptionにブランチ・PR一覧を記録
6. 結果報告

このスキルが呼び出された時点で、ユーザーにはPRを作成する意思がある。push・PR作成・issue本文の更新はこのフローの一部として進めてよく、途中で改めて許可を取り直す必要はない。ただし、未コミット変更のcommitやフロー外の操作は別の判断なので、必要になったらユーザーに確認する。

## 1. 前提確認

- `git branch --show-current` で現在のブランチを確認する。main/masterの場合はPRを作らず、作業用ブランチが必要なことをユーザーに伝える（ブランチ名にはissue番号を含める。例: `issue-123-description`）
- `git status` で未コミットの変更を確認する。PRに含めるべき変更が未コミットで残っていそうなら、コミットするかどうかをユーザーに確認する
- `gh pr view --json url,state,isDraft` で現在のブランチに既にPRがあるか確認する。openなPRが既にあれば新規作成はせず、そのPRを対象にステップ5（issueへの記録）だけを行う

## 2. 関連issueの特定

- ブランチ名からissue番号を抽出する（例: `issue-123-add-login` → 123）
- ブランチ名から特定できない場合は AskUserQuestion でissue番号を確認する。「関連issueなし」も選択肢に含める
- issueが特定できたら `gh issue view <番号> --json url,title,body` で存在確認と本文の取得をまとめて行う。存在しなければ番号を確認し直す
- 関連issueなしの場合、ステップ5はスキップする

## 3. push

- 未pushのコミットがあれば `git push -u origin <ブランチ名>` でpushする
- main/masterへの直pushは行わない

## 4. draft PRの作成

`gh pr create --draft` で作成する。本文はシェルのエスケープ事故を避けるため、一時ファイル（`mktemp`）に書いて `--body-file` で渡す。

### タイトル

日本語で、変更の目的が一目でわかるように書く。

### 本文の方針

PR descriptionで欲しいのはdiffの再説明ではなく、diffからは読み取りにくい文脈:

- 「どのファイルをどう変えたか」はdiffを見れば分かるので書かない
- PR descriptionの役割は、diffの代替ではなく、diffを読むための地図
- 「何をしたのか」より「なぜやったのか」「どういう判断をしたのか」を重視する

### 本文テンプレート

```markdown
## 概要

(この変更が必要になった背景・目的を1〜3文で)

関連issue: (issue URL)

## 文脈・判断

(diffから読み取りにくいことを書く: なぜこの方式にしたか、検討して採用しなかった代替案、
 レビュー時に注目してほしい箇所、影響範囲の注意点など。特になければセクションごと省略してよい)
```

関連issueがない場合は「関連issue: なし」と書く。

### 文体

- 日本語・平易な標準語で書く。口語スラング・比喩表現は使わない
- 変換例: 「データが着弾する」→「データが届く」、「デグレ」→「リグレッション」、「爆速」→「大幅に高速」、「バグを潰す」→「バグを修正する」、「メソッドが生える」→「メソッドが追加される」
- 迷ったら基準: 初見のレビュアーが文脈なしで一意に理解できる表現か

## 5. 親issueへの記録

関連issueが特定できた場合のみ行う。目的は、issueを見ればどのブランチ・PRで対応が進んでいるか一覧できる状態を保つこと。

1. ステップ2で取得したissue本文を一時ファイルに保存する
2. このissueに紐づくブランチを収集する:
   - 今回PRを作成した（または既存PRの）ブランチ
   - `gh issue develop --list <番号>` でissueにリンク済みのブランチ
   - `git branch -r --format='%(refname:short)'` のうち `issue-<番号>-` を名前に含むブランチ（`origin/` プレフィックスは除いてブランチ名として扱う）
3. 各ブランチに対応するPRを `gh pr list --head <ブランチ名> --state all --json url` で調べる
4. issue本文の `## PR` セクションを更新する:
   - 既にあれば、セクション内の既存行は保持したまま、一覧にないブランチの行だけ追記する
   - なければ本文末尾に新設する
   - 行の形式:

```markdown
## PR

- `issue-123-add-login`: https://github.com/owner/repo/pull/45
- `issue-123-fix-validation`: (PR未作成)
```

5. 更新後の本文を一時ファイルに書き、`gh issue edit <番号> --body-file <ファイル>` で反映する

注意:

- issue本文のPRセクション以外は一切変更しない。書き換え前に変更前後の差分（`diff <保存した元本文> <更新後本文>`）を確認し、PRセクションへの追記以外の変更が混ざっていないことを確かめてから反映する
- issue本文の取得と反映の間に他の編集が入る可能性は低いが、反映後に `gh issue view` で結果を確認する

## 6. 報告

- 作成したPRのURL
- issueに記録した内容（更新した場合はPRセクションの最終形）
- スキップした手順とその理由（例: 関連issueなしのためissue更新をスキップ）
