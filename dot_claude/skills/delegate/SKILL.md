# Supervisor/Worker 委譲モード

このスキルが呼び出されたら、以降のセッションではあなた（現在動作中のモデル）が監督役（supervisor）となり、実作業を下位の低コストsubagent（worker）へ委譲する体制で進める。

## 目的

上位モデルのトークンは高価で、usage limitも消費する。実作業のトークン消費を安く速いモデルへ移し、上位モデルの思考は計画・検証・判断に集中させる。これにより課金とusage limitを節約し、並列実行と高速モデルの活用で作業全体を速くする。品質は「監督役の指示の質」と「成果物の検証」で担保する。

## 監督役の責務分担

自分でやる（委譲しない）:
- 要件の解釈、ユーザーとの対話・確認
- タスク分解と実行計画の作成
- workerの選択と委譲プロンプトの作成
- worker成果物の検証・統合・最終判断
- セキュリティ判断、破壊的操作の判断、git commit/pushの判断
- ユーザーへの報告

workerに委譲する:
- コードベースの調査・探索（実装箇所の特定、影響範囲調査、仕様の読み解き）
- 実装（コードの作成・編集）
- 検証作業（テスト・リンタの実行、動作確認、レビューの下読み）
- ドラフト生成、機械的・決定的な変換（フォーマット統一、一括置換、データ整形）

例外 — 委譲しないほうが速い場合:
委譲には固定コストがかかる（自己完結プロンプトの作成、subagentの起動、成果物の検証）。1〜2ファイルの自明な編集や、Grep/Read数回で済む確認は自分でやるほうが速くて安い。目安: 委譲プロンプトを書く手間が作業そのものと同程度なら、委譲しない。

## worker一覧

| agent | モデル / effort | 速度・コスト | 能力 | 主な用途 |
|---|---|---|---|---|
| sonnet-xhigh | Sonnet / xhigh | 最速・最安 | 低 | 汎用の下請け（調査・実装・検証・変換） |
| light-implementer | Sonnet | 最速・最安 | 低（実装特化） | 承認済みの詳細な実装計画の忠実な実行 |
| opus-medium | Opus / medium | 中 | 中 | 一定の判断を伴う調査・実装 |
| opus-xhigh | Opus / xhigh | 最も遅い・高コスト | 高 | 難度の高い調査・実装・デバッグ |

注意: 現在のセッションモデルと同等以上のworker（例: セッションがOpus mediumのときのopus-xhigh）はコスト削減にならない。使ってよいのは、並列化またはコンテキスト分離（大量のファイル読み込みを監督役のコンテキストに入れない）が目的の場合だけ。

## worker選択 — 2軸で決める

委譲が成功するかは「指示の完全さ × workerの能力」で決まる。指示が完全であるほど低い能力で足り、指示に判断の余地が残るほど高い能力が必要になる。監督役が計画を具体化する時間を掛けるほど安いworkerに落とせる。このトレードオフを意識して、tierを上げる前に「指示を具体化すれば下のtierで足りないか」を先に検討する。

- 軸1: 事前指示の確実性 — 変更対象ファイル・具体的な手順・期待する成果物を、どこまで確定した状態で渡せるか
- 軸2: タスクの重さ — 設計判断・デバッグの難しさ、影響範囲の広さ、失敗時の手戻りの大きさ

| | 指示が具体的（手順・対象まで確定） | 指示に判断の余地が残る |
|---|---|---|
| 軽い（機械的・定型） | sonnet-xhigh | sonnet-xhigh |
| 中程度 | light-implementer（実装）/ sonnet-xhigh | opus-medium |
| 重い（設計・難デバッグを含む） | light-implementer（計画済み実装）/ opus-medium | opus-xhigh、または監督役自身 |

- デフォルトはまず sonnet-xhigh を検討し、不足する具体的な理由（判断の余地が大きい、影響範囲が広い等）を言えるときだけtierを上げる。
- light-implementer と sonnet-xhigh の使い分け: 監督役が作った詳細な実装計画を実行させるなら light-implementer（実装専用の規律を持つ）。調査・検証・変換など実装以外の作業、または計画がそこまで詳細でない実装は sonnet-xhigh。
- 「重い × 判断の余地あり」を委譲する前に、監督役が計画立案だけ自分で行い、「重い × 指示が具体的」に変換してから安いworkerに渡せないか検討する。これが最もコスト効率が良い分業になる。

## 委譲プロンプトの書き方

workerはこの会話の履歴も監督役の頭の中の計画も見えない。委譲プロンプトは自己完結させ、必ず以下を含める:

- 目的: なぜこの作業をするのか（背景を1〜2文で）
- 成果物形式: 何をどんな形式で返すか（ファイル変更 / パスと行番号のリスト / pass・failの報告 など）
- スコープ: 対象範囲と、やらないこと
- 参照パス: 読むべきファイル・ディレクトリ、参考になる既存の類似実装
- 検証方法: 実行すべきテスト・リンタ・型チェックのコマンド（分かっている場合）

NG: 「良い感じにやって」「最終判断して」のような丸投げ。判断はworkerの仕事ではない。判断の余地を残すなら、それはtierを上げる理由になる。

## 並列化

独立したタスクは、1つのメッセージで複数のworkerを同時にspawnする（調査をモジュール別に分割する、実装とテスト整備を分ける等）。ただし同じファイルを複数のworkerに同時編集させない。編集が衝突しうる場合は直列にするか、分割の単位を見直す。

## 受け取りと検証

worker出力は自己申告を信用せず、監督役が検証してから採用する。tierが低いworkerほど検証を厚くする。

- 調査結果: 結論を支える主要なパス・関数名・APIをRead/Grepで抜き取り確認する（全件でなくてよい）
- 実装: git diffで変更内容を読み、スコープ外変更の混入がないか確認する。テスト・リンタの実行結果を確認する（workerが実行していなければ監督役側で実行する）
- 検証で問題が見つかったら: 軽微なら監督役が直接修正、構造的な問題なら破棄して再委譲または自分で実装

## エスカレーション

workerが失敗・ブロックを報告したとき:

1. 原因が指示の曖昧さなら、指示を具体化して同じtierに再委譲する（SendMessageで同じworkerに追加指示すると、workerのコンテキストを維持したまま続行できる）
2. 原因が能力不足なら、一段上のtierに委譲する
3. 同じタスクで2回失敗したら委譲をやめ、監督役が自分で実施する。失敗の繰り返しはコストと時間の両方で逆効果になる

## 報告

ユーザーへの作業報告には委譲の内訳を含める: どのworkerに何を委譲したか、検証で修正・破棄した内容があればその要点。委譲の判断根拠まで長々と説明する必要はない。
