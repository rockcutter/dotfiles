# PR 作成

PR description の役割は diff の再説明ではなく、**diff を読むための地図**。
最適化するのは「不明点ゼロの網羅」ではなく「レビュアーが承認判断に至るまでの最短経路」。
レビュアーが3分で「なぜこの作業が存在し、何を承認しようとしているのか」を掴めるかを基準にする。
湧きうる疑問すべてに先回りで答えると、全読者に読むコストを課すわりに情報の重要度の階層が消える。
答えはレビューの質疑でも出せることを思い出す。

## 本文の基本形

```markdown
親issue: <URL>

## 背景・課題
issue の背景の要約(ほぼ転記でよい)。なぜこの作業が存在するのか。
最後を「そこで◯◯することにした。本 PR はその△△」と、
決定とこの PR の位置づけで締める。

## やったこと
この PR のスコープを短く。何の一部か(第一歩・お試し等)と、
何をやらないか(スコープ外の肝がどの PR / issue にあるか)を明示する。

## 検証
実行したコマンドと確認した事実。客観的・再現可能な形で書く。
```

セクションの追加は可(例: diff を見たら必ず湧く疑問に答える「実装上の判断」)。
ただし追加するたびに下の判断基準でふるいにかける。

## 書くときの判断基準

- **各段落に「これが無いとレビュアーは承認判断のどこで困る?」を自問する**。答えられない段落は削る
- **詳細説明の置き場所はコード内コメント > issue > PR 本文の優先順**。既にコードコメントや issue にある説明を PR 本文へ複製しない(複製は更新時にドリフトする)。PR からはリンクや一言の参照で足りる
- **期待値設定の一言を惜しまない**。「お試し」「第一歩」「肝は別 PR」のような一言は、説明の段落ひとつ分より読者の理解を進める
- **先回りの説明は「diff を見たら必ず湧く疑問」に限る**(自動生成ファイルの大量差分、無関係に見えるファイルを触った理由など)。「湧くかもしれない」程度の疑問はレビューの質疑に任せる
- **用語を導入しない**。定義セクションが必要になる語(比喩・チーム内スラング)は使わず、仕組みを一度説明したら、あとは「一緒に実行される」のような平易な日本語で言い直す
- **リスクを伴う情報(マージ順・デプロイ手順・切り戻し)は、そのリスクを持つ側の PR に書く**。関係ない側の PR へ複製しない

## タイトル

- リポジトリのマージ履歴(`git log --oneline -20`)を確認し、squash merge + conventional commits 運用なら `feat(scope):` 等の prefix を付ける(PR タイトルがそのまま commit message になるため)
- 「お試し」「第一歩」のような位置づけの語はタイトルに入れてよい。1語でレビューの期待値が決まる

## 手続き

1. ブランチを確認する(master / main への直 push 禁止。ブランチ名に issue 番号を含める)
2. commit・push の前にユーザーの許可を得る
3. `gh pr create --draft` で作成する。本文は日本語
4. 親 issue がある場合、issue description に「PR」セクションを追加し、各 PR のリンクと役割・マージ順を1行ずつ記載する
5. 文体は平易な標準語。比喩スラングを使わない。URL の前後に半角スペースを入れる

## 投稿前セルフチェック

- 背景を知らない人が最初の3文で「なぜやるか」に着地できるか
- diff を読めば分かることの再説明になっている段落はないか
- 「この PR でやらないこと」と、それがどこにあるかが一目で分かるか
